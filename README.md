# m1n3

A trustless Bitcoin mining pool built on Sui, with a DeepBook-based marketplace for trading miner hashpower.

m1n3 is two things in one protocol:

1. **A decentralized mining pool.** Shares are signed Sui transactions, reward accounting is on-chain, and the BTC payout destination is derived from a Sui object's UID rather than chosen by an operator. Because that destination has no private key controlled by anyone, both custody and claiming are trustless.
2. **A hashpower marketplace.** On top of the pool, DeepBook V3 provides liquidity for HashShare derivative markets and a buyer-template lane where buyers fund USDC orders that pay miners per share for hashing their templates.

Built on Sui, compatible with Stratum v1.

## Design

Conventional pools are custodial on the payout side: shares are counted in a private database, the operator assigns per-share value, and the operator's wallet pays out. m1n3 moves each of these steps on-chain.

| Custodial pool step | m1n3 equivalent |
|---|---|
| Database of accepted shares | Each accepted share is a `pool::submit_share` Sui tx signed by the miner |
| Operator decides per-share value | `RoundHistory.total_net_work` is frozen on-chain; a miner claims `net_work / total_net_work × batch.balance` |
| Operator chooses the BTC payout wallet | Coinbase output script is derived from a `HashiVault<BTC>` object UID (see below) |
| Operator can pause withdrawals | `claim_reward` is permissionless, gated only by the holder's `MinerWorkRecord` |

The reward invariants are enforced by Move's type system rather than by operator policy.

## Trustless payout address

The defining design choice is how the BTC reward address is selected: it isn't selected by anyone.

At pool initialization, anyone can create a shared vault:

```move
// contracts/sources/hashi_vault.move
public fun create_shared<HBTC>(_cap: &PoolAdminCap, ctx: &mut TxContext) {
    let vault = HashiVault<HBTC> { id: object::new(ctx), ... };
    let vault_id = object::uid_to_address(&vault.id);
    event::emit(VaultCreated { vault_id, derivation_path: vault_id });
    transfer::share_object(vault);
}
```

The vault's UID is a 32-byte address generated by Sui at object creation (a function of the tx digest and a creation counter). That UID is passed to Hashi's MPC committee as the derivation path. Per the BIP-340/341 scheme in `hashi-derive/src/main.rs`:

```
tweak       = HKDF-SHA3-256(master_x || vault_uid)
child_xonly = master_g ⊕ tweak                          (BIP-340)
output_key  = taproot_tweak(child_xonly, NUMS_internal) (BIP-341)
```

The resulting P2TR is baked into every coinbase output the stratum server constructs. The operator cannot reroute the subsidy, because the destination is derived from on-chain state, not configuration. The vault has no private key held by any party. When Hashi is live on mainnet, hBTC accrues to this object, and miners claim their slice directly from it via `hashi_rewards::claim_reward<BTC>` against a valid `MinerWorkRecord` — making both custody and claiming fully trustless.

See `docs/trustless-address.md` for the worked example.

## Architecture

```
                            Bitcoin Core
                                 │  getblocktemplate
                                 ▼
                          stratum-server :3333  ───── registers Template on Sui
                                 │  Stratum v1            (coinbase pays to the
ASIC ─┐                          │                        Hashi-derived P2TR)
      │  Stratum v1              │
      ▼                          │
miner-sidecar :3334 ─────────────┘
      │   signs every accepted share with the miner's Sui key
      ▼
m1n3 contracts on Sui
   │
   ├─ pool::submit_share       (owned objects — parallel)
   ├─ pool::open_round_…       (permissionless, gated by BlockFoundClaim)
   ├─ pool::finalize_round     (permissionless after window)
   ├─ hashi_pool::record_…     (operator publishes a CONFIRMED signet TX)
   ├─ hashi_rewards::open_and_fund_round_batch  (permissionless)
   └─ hashi_rewards::claim_reward<BTC>          (each miner claims their slice)
```

## Hashpower marketplace

Built on DeepBook V3, the marketplace lets buyers pay miners for hashing their block templates and provides liquidity for HashShare positions.

- **HashShare market** — DeepBook V3 routing on the `/marketplace` public tab.
- **Buyer-template lane** — buyers fund USDC orders that pay per share. Two order kinds:
  - `BuyerHashpowerOrder<QuoteT>` is buyer-bound: `submit_share_for_buyer_pay` asserts `template.owner == order.buyer`, so the order drains against any future template the buyer registers without a cancel/re-place cycle.
  - A legacy template-pinned order kind is retained for back-compat cleanup.

The `stratum-server` binary runs in either operator mode (admin cap set) or private-buyer mode (cap cleared, `SUI_ADDRESS` = buyer wallet, templates auto-register via `register_template_public` at 0.01 SUI/template). Both order kinds appear on `/marketplace?tab=private` with inline cancel / re-price / top-up actions.

## Requirements

| Layer | What | Notes |
|---|---|---|
| Bitcoin full node | Source of `getblocktemplate`; broadcasts found blocks | Mainnet for production, signet for the Hashi devnet integration. Pruning to ~50 GB is fine. Open RPC (8332 mainnet / 38332 signet) to the stratum-server. |
| Sui CLI + keypair | Signs template registrations, share submissions, round-close PTBs | `sui client switch --env mainnet`. Hold ≥ 5 SUI for gas plus 0.01 SUI per buyer-mode template registration. |
| Sui RPC endpoint | All PTBs route through it | Mysten's public fullnode works but is heavily loaded; a self-hosted or paid endpoint is recommended for production share traffic. |
| stratum-server | Stratum v1 on :3333; validates shares locally, registers each fresh Template on Sui | One Rust binary; operator or private-buyer mode via env. |
| miner-sidecar | Proxy on :3334 ASICs point at; each accepted share becomes a miner-signed Sui tx | Required when the ASIC can't hold a Sui keypair. |
| ASIC | Computes SHA-256d work | Any Stratum v1 device. Connect to `stratum+tcp://<sidecar-host>:3334`, username `<sui-address>.<worker>`. |
| Web dapp (optional) | Dashboard for templates / shares / rewards / marketplace | Next.js static export; `npm run build` from `web/`, deploy `out/`. |
| Storage | ~700 GB full `bitcoind` (~50 GB pruned), ~2 GB Rust builds, ~1 GB web build | |

Minimum solo setup: one `bitcoind` + one `stratum-server` + one `sidecar` + one ASIC. The same software scales to a multi-ASIC pool or a buyer-side rig.

A subsequent **v5** publish (`0x640abaf8…31a588b77`, published 2026-06-21, after the Overflow submission deadline) ships throughput improvements only — per-round share-dedup scoping (one `ShareDedup` per `(miner, round_id)` instead of per template, reducing per-share bootstrap txs), a `DerivedTemplate.round_id` field so the buyer-pay lane shares the same scoping, and a `MAX_MERKLE_BRANCHES` cap in `register_template_inner`. v5 is out of scope for the Overflow submission and changes no protocol semantics.

### Not on mainnet yet (gated on upstream)

The Hashi MPC bridge committee is not yet deployed to mainnet (Hashi is devnet-only today). The `hashi_pool` / `hashi_vault` / `hashi_rewards` modules are compiled into the mainnet package and `HashiRewardRegistry` is shared, but the `record_block_found → register_with_hashi → claim_reward<BTC>` pipeline is gated on a live Hashi committee that can custody the coinbase output. When Hashi lands on mainnet, the existing PTBs resolve against the mainnet vault with no contract changes.

## Quick start

```bash
# 1. Move contracts
cd contracts && sui move test --build-env testnet
sui client publish --gas-budget 600000000 --skip-dependency-verification

# 2. Stratum server (needs a running bitcoind)
cp .env.example .env  # fill in SUI_PACKAGE, POOL_OBJECT, BITCOIN_RPC, POOL_ADDRESS
scripts/start-stratum.sh

# 3. Web dapp
cd web && cp .env.example .env.local  # fill in NEXT_PUBLIC_*
npm install && npm run dev
```

ASICs connect to `stratum+tcp://<host>:3333` (operator path) or `stratum+tcp://<host>:3334` (trustless sidecar path).
