# m1n3

**A trustless Bitcoin mining pool that lives on Sui.**

Every share is a signed Sui transaction. Every payout is an on-chain claim.
The BTC reward address is a deterministic function of a Sui object's UID —
nobody chooses it, not even the operator.

## The problem

Bitcoin mining is a $30B/yr industry where the entire payout side is
custodial. A miner connects their ASIC to a pool, the pool counts shares
in a private MySQL table, the pool decides what each share is worth,
the pool's hot wallet pays out — and the only artifact a miner sees is a
dashboard the pool operator can rewrite on a whim. Foundry, AntPool, F2,
ViaBTC: all of them have, at one point or another, paused withdrawals,
miscounted shares, run "PPLNS" with the parameters undocumented, or
disappeared with funds.

m1n3 replaces every one of those custodial moments with an on-chain
counterpart on Sui:

| What pools do today | What m1n3 does |
|---|---|
| Database of accepted shares | Each accepted share is a `pool::submit_share` Sui tx signed by the miner |
| Operator decides per-share value | `RoundHistory.total_net_work` is frozen on chain; miners claim `net_work / total_net_work × batch.balance` |
| Operator chooses the BTC payout wallet | Coinbase output script is derived from a `HashiVault<BTC>`'s UID — see [the trick](#the-trick) |
| Operator can pause withdrawals | `claim_reward` is permissionless and gated only by the holder's `MinerWorkRecord` |
| "Trust us" | All four invariants are enforced by Move's type system |

## The trick

The single most important design choice in m1n3 is how the BTC reward
address gets picked. **It doesn't.**

When the pool is initialized, anyone can call:

```move
// contracts/sources/hashi_vault.move
public fun create_shared<HBTC>(_cap: &PoolAdminCap, ctx: &mut TxContext) {
    let vault = HashiVault<HBTC> { id: object::new(ctx), ... };
    let vault_id = object::uid_to_address(&vault.id);
    event::emit(VaultCreated { vault_id, derivation_path: vault_id });
    transfer::share_object(vault);
}
```

The vault's `UID` is a 32-byte address generated by Sui at object creation
time — a function of the tx digest and a creation counter. Nobody picks
it.

That `UID` is then fed to Hashi's MPC committee as the `derivation_path`.
Per the BIP-340/341 scheme implemented in `hashi-derive/src/main.rs`:

```text
tweak       = HKDF-SHA3-256(master_x || vault_uid)
child_xonly = master_g ⊕ tweak                        (BIP-340)
output_key  = taproot_tweak(child_xonly, NUMS_internal)  (BIP-341)
```

The resulting P2TR — `bc1pjn4qygxunmagm6maqux2jl68u98wsj72h2x7pk7gdkljgy8q9tvqfwxwmf`
on mainnet, encoded as `tb1pjn4q…sppx` on Bitcoin signet — is what the
stratum server bakes into every coinbase output it constructs. The
operator can never reroute the subsidy to a wallet they control because
**the destination is on-chain math, not a config setting.** The only way
those sats leave the vault is `hashi_rewards::open_and_fund_round_batch`
→ `claim_reward<BTC>` to a miner with a valid `MinerWorkRecord`.

That's the protocol.

See [`docs/trustless-address.md`](docs/trustless-address.md) for the
worked example and the proof line (in the running stratum log:
`Pool address script: 5120…0ad8`).

## Architecture

```text
                            Bitcoin Core
                                 │  getblocktemplate
                                 ▼
                          stratum-server :3333  ───── registers Template on Sui
                                 │  Stratum v1            (coinbase pays to the
ASIC ─┐                          │                        Hashi-derived P2TR)
      │  Stratum v1              │
      ▼                          │
miner-sidecar :3334 ─────────────┘
      │   signs every accepted share with the miner's Sui key
      ▼
m1n3_v4 contracts on Sui
   │
   ├─ pool::submit_share       (owned objects — parallel)
   ├─ pool::open_round_…       (permissionless, gated by BlockFoundClaim)
   ├─ pool::finalize_round     (permissionless after window)
   ├─ hashi_pool::record_…     (operator publishes a CONFIRMED signet TX)
   ├─ hashi_rewards::open_and_fund_round_batch  (permissionless, drains
   │                                              exactly record.amount_sats)
   └─ hashi_rewards::claim_reward<BTC>           (each miner claims their slice)
```

### Buyer-template lane

A second on-chain rail (V3 + V4 upgrades) lets buyers fund USDC orders that pay miners per share for hashing *their* block templates:

- **V1 `HashpowerBuyOrder<QuoteT>`** — pinned to a single Template id. Legacy; goes stale on every buyer re-publish. Kept for back-compat cleanup.
- **V2 `BuyerHashpowerOrder<QuoteT>`** — buyer-bound. `submit_share_for_buyer_pay` asserts `template.owner == order.buyer` so the same order drains on any future template the buyer registers — no cancel/re-place cycle.

The `stratum-server` binary doubles as the buyer's full node: clear `POOL_ADMIN_CAP` and set `SUI_ADDRESS` to the buyer wallet, and templates auto-register via `register_template_public` (0.01 SUI fee per template, `Template.owner = buyer`). Both order kinds appear side-by-side on `/marketplace?tab=private` with a V1/V2 chip and inline cancel / re-price / top-up actions for the order's buyer. See [`CLAUDE.md`](./CLAUDE.md#7-buyer-template-lane) for the full surface (entries, ops binaries, `ProtocolMPCConfig`).

## Quick start

```bash
# 1.  Move contracts
cd contracts && sui move test --build-env testnet
sui client publish --gas-budget 600000000 --skip-dependency-verification

# 2.  Stratum server (needs a running bitcoind)
cp .env.example .env  # fill in SUI_PACKAGE, POOL_OBJECT, BITCOIN_RPC, POOL_ADDRESS
scripts/start-stratum.sh

# 3.  Web dapp
cd web && cp .env.example .env.local  # fill in NEXT_PUBLIC_*
npm install && npm run dev
```

ASICs connect to `stratum+tcp://<host>:3333` (operator path) or
`stratum+tcp://<host>:3334` (trustless sidecar path).

## Sui Overflow 2026

Submitted to the **DeFi & Payments** core track. The protocol's core
claim — programmable, on-chain Bitcoin mining-pool payouts where the
BTC recipient itself is a function of Sui state — is a programmable
payment primitive. See the [submission](https://mystenlabs.notion.site/defi-payments-problem-statement)
for context.

Also entering the **DeepBook track** with the mining-revenue hedge in
[`docs/hedge.md`](docs/hedge.md): an end-to-end integration of
DeepBook Predict that lets miners synthesize a put on their expected
BTC revenue using the live SVI vol surface. The `/hedge` route on the
dapp signs a single PTB chaining `predict::create_manager` +
`predict_manager::deposit<DUSDC>` + N×`predict::mint_range<DUSDC>`,
and shows a Monte-Carlo proof that hedged variance < unhedged
variance and that the 5th-percentile revenue lifts strictly.

A **confidential-transfers track** integration is partially landed
in [`docs/otc.md`](docs/otc.md): on devnet, where m1n3, Hashi, and
[MystenLabs/confidential-transfers](https://github.com/MystenLabs/confidential-transfers)
all coexist, the new `m1n3_confidential_otc` Move escrow lets a
miner and an institution settle a `Coin<HS_NNN>` ↔ `Coin<DUSDC>` swap
in one atomic PTB. Post-settle the institution can confidentially
wrap their received HashShare via the TS SDK, and later
`hash_share::redeem` it for real hBTC. Phase A's escrow object
publishes the trade size on chain; Phase B (on-chain confidential pay
leg + NIZK equality proof) is documented and out of this PR.
